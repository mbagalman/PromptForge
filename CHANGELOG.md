# Changelog

This file records meaningful changes to prompts in this repository. Bump rules and entry format are defined in [`CONTRIBUTING.md`](CONTRIBUTING.md#versioning-and-changelog).

The `version` field in each prompt's frontmatter is the source of truth for the *current* version; this file is the source of truth for *what changed between versions*.

## 2026-05-21

- **New `development/` directory** — adds a new top-level home for engineering-process prompts (workflow suites, retrospectives, code-review prompts that don't fit narrow `coding/`, etc.). The first occupant is `development/coding-workflow-suite/` — the Structured Coding Workflow Suite from the ROADMAP. Distinct from `coding/`, which remains for per-language code-quality tools that operate during or after implementation; `development/` is for engineering-process prompts that operate before or around implementation.
- **New `development/coding-workflow-suite/` directory ships with six prompts** — an end-to-end prompt sequence that forces disciplined process before any code is written: an orchestrator, four artifact-producing stages, and a single-decision recorder. Initial canonical version 1.0.0 for all six, status `experimental` (first folder in the repo to ship at experimental — every other folder shipped at `stable` because its prompts had been used before publication; the workflow suite has no prior field use, and the controlled vocabulary supports `experimental` with a clear promotion path to `stable` after the chain is exercised at volume). A seventh prompt (`agents-md-generator.md`) for translating the Implementation Plan and resolved ADRs into an AGENTS.md / CLAUDE.md / SKILL.md file is in development and will land in a forthcoming update. The suite is grounded in a state-machine specification (recorded in working notes) with explicit approval gates owned by the orchestrator, an ADR loop between Tech Spec Draft and Tech Spec Revision, a suite-wide input-type taxonomy (`supplied` / `inferred` / `user-confirmed`) that makes mid-chain and standalone runs auditable, and Router / Integrated orchestrator modes.
  - `development/coding-workflow-suite/workflow-orchestrator.md` — initial canonical version 1.0.0. Drives the chain. Phase 1 asks the size-scaling question and picks one of three chains: **Small** (`prd.md` → `implementation-plan.md`), **Medium** (`prd.md` → `tech-spec.md` → `implementation-plan.md`), **Large** (`brd.md` → `prd.md` → `tech-spec.md` Draft → `adr.md` loop → `tech-spec.md` Revision → `implementation-plan.md`). Operates in user-selectable Router mode (default; names the next stage to run, never writes artifacts) or Integrated mode (produces the next artifact inline by applying the stage prompt's full rules, with a header disclosing which stage's contract was applied so the output is auditable). Enforces approval gates between stages, supports overrides with explicit assumption logging in the next downstream stage's Assumptions table, and treats artifacts as the workflow's only state (no orchestrator state file). Tags: `coding-workflow`, `orchestration`.
  - `development/coding-workflow-suite/brd.md` — initial canonical version 1.0.0. Interactive five-exchange Business Requirements Document elicitation. Captures the business case (situation, stakeholders, outcomes, success criteria, constraints, non-goals, risks, open questions) before any product or technical work begins. Load-bearing rules: *outcome first, solution later* (redirects when users propose features instead of outcomes); *at least one measurable success criterion is required* (scope-stops the session when the user cannot articulate one); *stakeholder accountability is non-negotiable* (three-role taxonomy — primary decision-maker / affected groups / sign-off owners — no fabricated stakeholders); BRD–PRD–Tech Spec boundary discipline with concrete examples in *Constraints*. Defers KPI-decomposition depth to `kpi-architect.md`. BRD is net-new only (retrofits are framed around what the new initiative unlocks for the business). Tags: `coding-workflow`, `brd`.
  - `development/coding-workflow-suite/prd.md` — initial canonical version 1.0.0. Translates the BRD's *why* into a user-facing *what*. Four-phase elicitation (BRD intake and scope confirmation → personas and capabilities → functional and non-functional requirements → out-of-scope and artifact production). Load-bearing rules: *personas are named, never invented* (asks rather than synthesizing from stakeholder language); *functional requirements describe user-visible behavior* (rewrites "the system shall…" sentences); *every requirement has an acceptance criterion* (rework or split if missing); *out-of-scope is a real section, not a placeholder*; non-functional requirements enforced per-dimension (performance / reliability / security / accessibility / compliance each has requirements or an explicit "none required"). Implementation creep redirected to Tech Spec. Tags: `coding-workflow`, `prd`.
  - `development/coding-workflow-suite/tech-spec.md` — initial canonical version 1.0.0. Two-mode Technical Specifications prompt. **Draft mode** produces a first-pass system design (problem statement, system context diagram, component breakdown, data model, interfaces, failure modes and reliability targets, security, observability, cross-cutting concerns) and surfaces every architecturally-significant decision as a `Q-NNN`-tagged entry in *Open architectural questions*. **Revision mode** consumes the Draft plus resolved ADRs and produces a Final Tech Spec where every open question is marked `Resolved (see ADR-NNN)` or, under explicit override, `Unresolved — carried as Implementation Plan risk`. Mode detection by presence of resolved ADRs in the input. Two distinct artifact files (`tech-spec-[project]-draft.md` and `tech-spec-[project]-final.md`) for traceability across the ADR loop. Load-bearing rules: *surface decisions; do not bury them*; *refuse to produce a Final Tech Spec with unresolved open questions* absent the explicit override; *component decomposition by responsibility, not by layer*; *reliability targets attach to failure modes*; retrofit mode demands a current-architecture summary before producing design content. Tags: `coding-workflow`, `tech-spec`.
  - `development/coding-workflow-suite/adr.md` — initial canonical version 1.0.0. Single-decision Architecture Decision Record in the canonical Michael Nygard format (Title / Status / Context / Decision / Consequences), plus *Options considered*, *Related ADRs*, and *Assumptions and inferred inputs* sections. One decision per run; multi-decision requests pause and chain. Status vocabulary: `proposed` / `accepted` / `superseded by ADR-MMM` / `deprecated` (extends Nygard's three with `deprecated` and the parameterized `superseded by ADR-MMM` form). Every ADR's header carries an `Originating question-id` field linking it back to the Tech Spec Draft's `Q-NNN` entry (or the literal `user-supplied` for standalone runs); this is what the Tech Spec Revision pass uses to close out open questions programmatically. Reversibility (Type 1 / Type 2 per Bezos's framing) must be named in Consequences; ≥2 options required. Decision-prep mode for not-yet-made decisions produces an ADR with Status `proposed` and Decision / Consequences blank. Tags: `coding-workflow`, `adr`.
  - `development/coding-workflow-suite/implementation-plan.md` — initial canonical version 1.0.0. Phased work breakdown with deterministic approval gates between phases. Five-exchange flow (framing against the Tech Spec → phase shape → gates and estimates → agent execution boundaries → risks and rollout). Load-bearing rules: *every phase has at least one approval-gate criterion*; *every gate criterion is deterministic* (refuses "looks good"-style gates); *every estimate carries a confidence band* (no bare point estimates); *Agent execution boundaries table is required, not optional* (the forthcoming `agents-md-generator.md` consumes it verbatim); *boundary entries name verbs and surfaces* (refuses "can make code changes"-style abdications). Single project-level *Agent execution boundaries* table with three classes (`allowed without approval` / `requires approval` / `prohibited`) and three columns (Action class / Examples / Rationale). Supports the override path from the Tech Spec where unresolved architectural questions carry forward as Risk register entries. Tags: `coding-workflow`, `implementation-plan`.

- **New `data-analysis/kpi-architect.md`** — initial canonical version 1.0.0. Interactive five-phase conversation (Scope → Objectives & framework → KPI selection → DuPont-style decomposition → Spec) that produces a written KPI specification with each KPI decomposed into the operational levers that mathematically drive it. Grounded in five established performance-measurement frameworks, each doing a distinct job: **Balanced Scorecard** (Kaplan & Norton, 1996) for perspective coverage across financial / customer / internal-process / learning-and-growth dimensions; **Spitzer's criteria for effective measures** (2007) as a six-point per-candidate filter (predictive, actionable, accurate, simple, strategy-aligned, gaming-resistant); **Grove's paired-metrics rule** (*High Output Management*) requiring every output-volume or speed metric to pair with a quality or cost counter-metric; **Marr's Key Performance Questions** as the background principle that KPIs derive from the strategic question they answer; **DuPont decomposition** (1920s, originally ROE = NPM × Asset Turnover × Equity Multiplier) as the structural mechanic for the supporting-metric tree. Worked decomposition library covers ROE, e-commerce revenue (Sessions × Conversion × AOV), SaaS ARR growth (additive: New + Expansion − Churn − Contraction), LTV, GMV, OEE, sales win rate (stage-conversion product), and pipeline coverage (ratio). Three load-bearing headline disciplines: **3–7 KPI ceiling** (subtraction over addition; pad-the-list pressure resisted), **decomposition must be mathematical** (each factor tree reconstructs the KPI under product / sum / ratio; "conceptually related drivers" are explicitly distinguished from a true decomposition and reworked or dropped), **pair output metrics with counter-metrics** (Grove's rule applied in Phase 3 and carried into the final spec, with the gaming pattern named when the user declines pairing). Cross-references to the rest of the data-analysis collection: redirects visualization requests to `dashboard-designer.md`, statistical-cause questions to `data-analyst.md`, forecasting to `predictive-modeling.md`, and executive translation to `executive-insights.md`.
